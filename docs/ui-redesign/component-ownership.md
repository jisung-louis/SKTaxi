# Component Ownership Policy

작성일: 2026-03-10

## 1. 목적

UI 2.0 구현 중 기존 컴포넌트를 어떻게 다룰지 기준을 고정한다.

## 2. 기본 원칙

- 기존 컴포넌트는 당장 전면 교체하지 않는다.
- 기존 컴포넌트는 레거시 자산으로 유지한다.
- UI 2.0 구현 시 필요한 컴포넌트는 v2 전용으로 새로 만든다.
- 기존 컴포넌트는 참고자료와 아직 v2.0으로 전환되지 않은 화면 호환용으로 남겨둔다.

## 3. 구현 원칙

### 3.1 신규 구현 우선

- 상위 화면과 하위 화면의 v2.0 UI는 기존 컴포넌트 패치보다 신규 컴포넌트 작성이 기본값이다.
- 기존 컴포넌트의 props 구조나 내부 스타일 제약에 맞추기 위해 v2.0 설계를 훼손하지 않는다.

### 3.2 레거시 유지

- 아직 v2.0으로 개편되지 않은 화면은 기존 컴포넌트를 계속 사용해도 된다.
- 레거시 화면 안정성을 위해 기존 컴포넌트의 제거/대규모 수정은 전체 전환 전까지 보류한다.

### 3.3 참고자료 활용

- 기존 컴포넌트는 도메인 동작, 데이터 shape, 기존 상호작용을 이해하는 참고자료로 활용한다.
- 단, 시각 구조나 스타일은 v2.0 구현의 기준이 아니다.

## 4. 폴더/소유권 방향

- 기존 공용 컴포넌트는 legacy 영역으로 간주한다.
- v2.0 공용 컴포넌트는 새 foundation/primitive 계층으로 분리한다.
- screen-specific v2 컴포넌트와 공통 v2 primitive를 혼합하지 않는다.

## 5. 제거 시점

- Campus / Taxi / Notice / Community / My 및 관련 하위 화면이 모두 v2.0 기준으로 전환된 뒤 제거를 시작한다.
- 제거 대상은 아래를 포함한다.
  - 더 이상 import되지 않는 legacy screen
  - legacy 전용 component
  - 구 토큰 직접 참조 코드
  - 구 navigation 구조에 묶인 화면/파일

## 6. 제거 원칙

- 제거는 한 번에 하지 않고 리뷰 가능한 작은 단위로 나눈다.
- runtime 교체와 legacy 삭제는 가능하면 같은 PR에 섞지 않는다.
- 제거 전에는 참조 여부를 반드시 확인한다.

## 7. Codex 실행 규칙

- v2.0 구현 중 기존 컴포넌트를 억지로 재사용하려 하지 않는다.
- 먼저 `새 v2 primitive가 필요한가`를 판단한다.
- 기존 컴포넌트 수정은 레거시 화면 유지에 필요한 최소 범위로 제한한다.
