# Token Migration Strategy

작성일: 2026-03-10

## 1. 목적

UI 2.0 구현 중 디자인 토큰을 어떤 방식으로 도입할지 기준을 고정한다.

## 2. 결론

- 기존 토큰을 즉시 덮어쓰지 않는다.
- v2 토큰을 별도로 만든다.
- 새 화면과 새 컴포넌트는 v2 토큰만 사용한다.
- 기존 화면은 전체 전환 완료 전까지 기존 토큰을 유지한다.

## 3. 왜 이렇게 가는가

- 현재 앱에는 아직 v2.0으로 전환되지 않은 레거시 화면이 남아 있다.
- 기존 토큰을 바로 바꾸면 레거시 화면까지 동시에 흔들린다.
- 이번 리뉴얼은 단계적 전환이므로 `레거시 유지 + v2 병행`이 작업 범위를 가장 잘 통제한다.

## 4. 적용 원칙

### 4.1 레거시 토큰 유지

- 기존 `colors`, `typography` 계열은 레거시 화면 호환용으로 유지한다.
- 전체 전환 전까지 레거시 토큰의 의미를 v2 기준으로 재해석하지 않는다.

### 4.2 v2 토큰 별도 관리

- v2 색상, 타이포, spacing, radius, shadow는 별도 계층으로 만든다.
- 새 primitive와 새 상위 화면은 v2 토큰만 참조한다.
- 새 화면에서 legacy token을 직접 참조하지 않는다.

### 4.3 혼합 사용 금지

- 하나의 v2 화면 안에서 legacy token과 v2 token을 섞지 않는다.
- 하나의 v2 공통 컴포넌트 안에서 hard-coded color와 token을 섞지 않는다.

### 4.4 임시 브리지 최소화

- 레거시 화면을 살리기 위한 브리지나 alias는 최소화한다.
- bridge가 필요하면 임시 성격임을 명시하고, 전체 전환 후 제거 대상으로 기록한다.

## 5. 사용 규칙

- Legacy 화면: legacy token 사용 가능
- v2 primitive: v2 token만 사용
- v2 상위 화면: v2 token만 사용
- 하위 화면이 v2로 전환되면 해당 화면도 v2 token으로 전환

## 6. 제거 시점

- Campus / Taxi / Notice / Community / My 및 관련 depth 2+ 화면이 모두 v2.0으로 전환된 뒤 legacy token 제거를 시작한다.
- token 제거는 runtime 전환과 분리된 정리 단계로 다룬다.

## 7. Codex 실행 규칙

- 새 UI를 구현할 때 기존 token 파일을 먼저 확장하려 하지 않는다.
- 먼저 v2 token 계층이 있는지 확인하고, 없으면 v2 계층부터 만든다.
- 레거시 화면을 깨지 않기 위해 legacy token 수정은 최소 범위로 제한한다.
